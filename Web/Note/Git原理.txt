Git的原理，
相同内容，永远只存一次，哈希去重。
只存修改过的文件，没改的只打个勾。
自动压缩所有内容。 HTML\CSS\JS、压缩率能达到70-90%

Git快照和普通压缩包对比：
增量存储，只存修改过的部分，每次都完整复制所有文件。
Git 能精确对比两个版本之间，哪一行代码改了。
可以多人同时修改，自动合并。
只要有一人有仓库，所有版本都在。
刚提交时，每个人间单独压缩
仓库变大后，把相似的文件打包，只存一个完整文件，其他的存差异片段（不同的代码）

不管是.md、.pdf、.jpg、.exe 还是任何格式的文件，Git 都会遵循这 3 条铁律：
只要内容完全相同，不管叫什么名字、放在哪个文件夹，永远只存一份（哈希去重）
刚提交时，每个文件单独用 zlib 压缩，存成独立的松散对象
仓库变大后，Git 会自动打包，尝试找相似文件做行 / 字节级别的 Delta 增量压缩

一、Git 最擅长的文件：纯文本文件（效果拉满）
包括：
.md、.html、.css、.js、.txt、.py、.java 等所有代码和纯文本文档
处理效果：
✅ 哈希去重：完美
✅ zlib 压缩：效果极好（压缩率 70%-90%）
✅ Delta 增量压缩：效果爆炸

1. 文件名和扩展名完全不影响
把一个README.md改名为README.pdf，Git 会认为是同一个文件，只存一份。Git 只看内容，不看名字、扩展名、位置。

2. Git 不适合存大的二进制文件
如果你的项目里有很多大于 10MB 的图片、视频、安装包，不要直接提交到 Git：
仓库会迅速膨胀到几 GB 甚至几十 GB
克隆、拉取、推送速度会变得极慢
即使你后来删掉这些文件，它们还会永远存在于历史版本里

3. 大二进制文件的正确处理方式
用 Git LFS（Large File Storage）：
Git LFS 会把大文件存在单独的服务器上
Git 仓库里只存一个指向大文件的指针
